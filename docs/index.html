AWS MFA Incident Response Simulator

Detection, Investigation, and Assisted Remediation

Problem Statement

MFA failures are noisy, ambiguous, and operationally expensive.

When users report "can't log in," operations teams face a diagnostic challenge: Is this a token expiration? Rate limiting? A policy misconfiguration? AWS CloudTrail emits events, but doesn't explicitly label root causesâ€”you must infer patterns from event structure.

This project demonstrates production-grade incident response workflows: detecting patterns in CloudTrail, correlating signals, and applying operational judgment about what to automate versus what requires human intervention.

Detection Logic

The system detects three distinct MFA failure patterns using CloudTrail event analysis:

1. MFA Authentication Failure (Consistent with Token Expiration)

Detection Signal

ConsoleLogin event with:

  • additionalEventData.MFAUsed = "No"
  • errorMessage present (typically "Failed authentication")
  • Optional correlation: Prior successful MFA login within 15-minute window

Why "consistent with" matters: AWS doesn't emit "token expired" explicitly. The patternâ€”failed login with MFAUsed=No, especially after a recent successful MFA sessionâ€”is consistent with token expiration, but could also indicate user error or clock drift. The language reflects operational uncertainty.

2. Rate Limiting / Account Lockout

Detection Signal

Pattern: 5+ failed ConsoleLogin events within 60-second window

  • Same userIdentity.userName
  • Same sourceIPAddress
  • All events have errorMessage

Detection method: Rate limiting is detected via CloudWatch Insights queries (post-hoc analysis) rather than real-time EventBridge rules, as it requires aggregating events over a time window. The pattern can also be simulated for testing.

When detected, this triggers automated incident tracking and alerting. After a cooldown period (5 minutes default), the responder Lambda updates the incident status to RESOLVED and sends a notificationâ€”but does not modify IAM or unlock accounts.

3. Policy Mismatch (MFA Present but Action Denied)

Detection Signal

AccessDenied error with:

  • errorCode = "AccessDenied"
  • userIdentity.sessionContext.attributes.mfaAuthenticated = "true"
  • Action attempted visible in requestParameters

This indicates a policy condition mismatchâ€”the user authenticated with MFA, but the IAM policy condition (e.g., aws:MultiFactorAuthPresent) isn't being satisfied. Common causes include using StringEquals instead of Bool for boolean conditions.

Detection Architecture: The system operates in dual modeâ€”it can process real CloudTrail events from EventBridge (detector mode) or accept synthetic test payloads (simulator mode). The Lambda automatically detects which mode based on event structure.

Runbook Excerpt

Rate Limiting / Account Lockout

HIGH Assisted Remediation

Step 1: Quantify the failure pattern

Query CloudWatch Insights for multiple failed ConsoleLogin attempts:

fields @timestamp, userIdentity.userName, sourceIPAddress, errorMessage
| filter eventName = "ConsoleLogin"
| filter ispresent(errorMessage)
| stats count(*) as failure_count by userIdentity.userName, sourceIPAddress, bin(5m)
| filter failure_count >= 5
| sort failure_count desc
Step 2: Analyze source IP

Determine if this is legitimate user error or potential attack. Check:

  • Is source IP from known corporate range?
  • Is user agent consistent with browser?
  • Geographic location reasonable?
Resolution: Assisted Remediation

The responder Lambda handles this scenario:

  1. Detects incident is past cooldown period (5 minutes default)
  2. Updates incident status in DynamoDB to RESOLVED
  3. Sends SNS notification that lockout has cleared
  4. Logs resolution time for metrics

What it does NOT do:

  • Does not modify IAM user
  • Does not unlock any account
  • Does not reset credentials

Design Philosophy

Why Assisted Remediation

This system demonstrates constrained automationâ€”automating state tracking and alerting, but not destructive actions. This reflects production operational judgment:

  • Blast radius control: Updating DynamoDB status is low-risk; modifying IAM policies or unlocking accounts requires safeguards
  • Remediation loops: Full automation can create feedback loops (e.g., auto-unlocking an account under attack)
  • Audit trail: All actions are logged; human review is required for policy changes

Why No IAM Mutation

The responder Lambda explicitly avoids modifying IAM:

  • Rate limiting is a security control working as intendedâ€”AWS handles the unlock automatically
  • Policy mismatches require policy review, not programmatic fixes
  • MFA device issues require user self-service or verified admin intervention

Detection vs. Analysis Separation

  • EventBridge: Real-time detection (low latency, pattern matching)
  • CloudWatch Insights: Post-hoc analysis (investigation, correlation, historical queries)
  • Dashboard: Situational awareness (volume trends, resolution metrics)

This separation allows each tool to serve its optimal use case without conflating concerns.

What This Is / Is Not

What This Is

  • Detection system: Identifies MFA failure patterns in CloudTrail events
  • Incident tracking: Stores incidents in DynamoDB with full event context
  • Alerting pipeline: SNS notifications for response teams
  • Assisted remediation: Automated state updates for rate limiting scenarios (after cooldown)
  • Investigation tooling: CloudWatch Insights queries and dashboard for analysis
  • Operational documentation: Runbooks with detection signals and resolution steps

What This Is Not

  • Not a SaaS product: This is infrastructure-as-code for incident response workflows
  • Not a public API: Lambda functions are deployed in your AWS account, not exposed publicly
  • Not an IAM management tool: Does not modify IAM users, policies, or MFA devices
  • Not a security scanner: Focuses on operational response, not vulnerability assessment
  • Not a replacement for SIEM: Complements existing security tooling with AWS-native detection
  • Not fully automated remediation: Only rate limiting scenarios have assisted remediation; others require manual intervention

Technical Details

Architecture Components

  • Lambda Functions: Dual-mode simulator/detector (Python 3.11) and responder for assisted remediation
  • EventBridge Rules: Pattern matching for ConsoleLogin failures and AccessDenied events
  • DynamoDB: Incident storage with TTL (7 days default)
  • SNS: Alert notifications for incident detection and resolution
  • CloudWatch: Metrics, logs, Insights queries, and 4-widget dashboard

Detection Patterns

EventBridge rules match:

  • source = ["aws.signin"] with eventName = ["ConsoleLogin"] and errorMessage present
  • errorCode = ["AccessDenied", "UnauthorizedAccess"] from IAM, S3, EC2, STS
